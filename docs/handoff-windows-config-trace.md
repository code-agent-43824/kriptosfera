# Handoff (Windows): диагностика загрузки config.ini у CryptoPro CSP Lite / Mini CSP

> Запускается в локальном контейнере Claude Code на Windows-машине разработчика.
> Есть `git` и `GH_TOKEN` для пуша. Задача — **диагностическая** (troubleshooting),
> не модификация ПО.

## 0. Контекст и границы (прочитать первым)

Это **авторизованная инженерная диагностика интеграции** легально приобретённого
лицензионного продукта **КриптоПро CSP Lite / CAdES Browser Plug-in**, который
владелец имеет право распространять, установленного **на собственной тестовой
машине владельца**. Цель — **выяснить, из какого пути продукт читает свой
конфигурационный файл `config.ini`**, чтобы корректно настроить его в составе
собственного продукта (Kriptosfera). Это обычная работа инженера техподдержки:
наблюдение за тем, какие файлы/ключи реестра открывает приложение, и проверка
рабочей конфигурации.

**В объёме задачи только безвредные, наблюдательные действия:**
- запуск штатной утилиты самого вендора (`cpconfig.exe`) из комплекта;
- временное копирование **собственного** конфигурационного файла вендора
  (`config.ini`) в разные каталоги-кандидаты и проверка, влияет ли это (с
  возвратом всё как было);
- наблюдение за доступом приложения к файлам/реестру штатным диагностическим
  инструментом Microsoft (Process Monitor / Sysinternals);
- чтение официальной документации вендора.

**Вне объёма (НЕ делать, это и не требуется):** модификация/патч бинарей,
дизассемблирование, обход проверок целостности/подписи, извлечение приватных
ключей, любые действия против чужих систем. Если какой-то шаг вдруг ощущается как
обход защиты или анализ уязвимостей — **остановись и спроси владельца**; по
задумке здесь этого нет, всё сводится к наблюдению и штатной настройке.

Терминология в работе и в записях — «troubleshooting / diagnose / observe config
loading / which path the app reads», а не «reverse / crack / bypass / exploit».

## 1. Что уже установлено (чтобы не повторять)

Зафиксировано фактами (см. `docs/minicsp-snapshots/` и
`docs/cryptopro-csp-lite-plan.md`):

- На чистой машine без системного CSP, с плагином `ADDMINICSP=1` и нашей
  страницей с ранним флагом `EnableInternalCSP`, в обычном Chrome:
  «Криптопровайдер не загружен», `About.CSPName(80)=0x80090017`, «получить
  тестовый сертификат» → `0x80090019` (`NTE_KEYSET_NOT_DEF`). **С вставленным
  Рутокеном (сертификат в формате CryptoPro) — без изменений.**
- `ADDMINICSP=1` **не пишет** ветку `Crypto Pro` в реестр (снапшоты подтвердили).
- Эталон на Linux: те же CryptoAPI-вызовы перечисляют провайдеров **из
  `config64.ini`**; если конфиг недоступен — **ровно `0x80090017`**.

**Живая гипотеза:** на Windows `capi20`/`cpcspi` (внутри `nmcades.exe`) **не
находит/не читает свой `config.ini`** (ищет не в том каталоге), и поэтому
провайдер «не определён» — та же ошибка, что на Linux без конфига. Нужно
**наблюдением** установить, по какому пути приложение ждёт `config.ini`.

Путь к Mini CSP: `C:\Program Files (x86)\Crypto Pro\CAdES Browser Plug-in\Mini CSP\`
(там лежат `cpconfig.exe`, `capi20.dll`, `cpcspi.dll`, `config.ini`).

## Шаг 1 — чёрный ящик: где читается config (без трассировки, самый чистый)

Запусти готовый скрипт (он уже в репозитории):
```powershell
./tools/windows/reverse-enum-test.ps1
```
Он прогоняет штатную `Mini CSP\cpconfig.exe -defprov -view_type` из самой папки
`Mini CSP` и из `C:\Windows`, плюс системные виды (`certutil`, advapi32). Это
сравнивает «вид со стороны CryptoPro-конфига» и «вид со стороны системного
реестра».

Затем — проверка расположения конфига **копированием собственного `config.ini`**
(каждый вариант по очереди, после каждого — вернуть как было; в Program Files
нужны права администратора, это временная диагностическая копия):
1. копию `Mini CSP\config.ini` положить в родительскую папку рядом с
   `nmcades.exe` (`…\CAdES Browser Plug-in\config.ini`);
2. перезапустить вкладку с нашей `internal-csp` страницей, посмотреть, изменился
   ли статус «Криптопровайдер …»;
3. удалить копию; повторить для других кандидатов, если будет смысл (например
   рабочий каталог процесса).

Зафиксируй в `findings-step1.md`: при каком расположении (если вообще) провайдер
начинает грузиться. Если `cpconfig` из `Mini CSP` перечисляет провайдеры (тип
80), а из `C:\Windows` — нет, это уже показывает, что конфиг резолвится
относительно текущего каталога.

## Шаг 2 — наблюдение доступа к файлу/реестру (Process Monitor)

Только если шага 1 недостаточно. Process Monitor — штатный диагностический
инструмент Microsoft Sysinternals; используем его как инженер поддержки, **только
для наблюдения**.

1. Запусти Process Monitor. Фильтры (Include):
   - `Process Name` is `nmcades.exe`;
   - и в отдельном прогоне добавь `Path` contains `config` (ловит
     `config.ini`/`config64.ini`).
2. Для проверки реестрового пути — отдельный прогон с `Operation` begins with
   `Reg` и `Path` contains `Crypto Pro`.
3. Начни захват, открой нашу `internal-csp` страницу, дождись статуса/нажми
   «получить тестовый сертификат», останови захват.
4. Сохрани **только релевантные строки** (по `config.ini` и по
   `…\Crypto Pro\…\AppPath`) экспортом в CSV → положи в
   `docs/minicsp-snapshots/windows-config-trace/procmon-config.csv` (можно
   вручную отфильтровать; полный трейс не нужен).

Что выписать в `findings-step2.md` (колонка `Result` ключевая):
- есть ли `CreateFile`/`QueryOpen` на `…\Mini CSP\config.ini` и каков результат:
  `SUCCESS` (нашёл — по какому полному пути) или `NAME NOT FOUND`/`PATH NOT FOUND`
  (искал — и путь покажет, ГДЕ он ждал файл);
- ищет ли он `config.ini` в «неправильных» местах (рядом с `nmcades.exe`, в
  `System32`, в рабочем каталоге) с `NAME NOT FOUND`;
- запрашивает ли `RegQueryValue` на
  `HKLM\SOFTWARE\[WOW6432Node\]Crypto Pro\Cryptography\CurrentVersion\AppPath` и
  с каким результатом.

## 3. Как трактовать (таблица решений)

| Наблюдение | Вывод |
| --- | --- |
| `cpconfig` из `Mini CSP` перечисляет 80, из `C:\Windows` — нет | конфиг резолвится по cwd/модулю; `nmcades.exe` (запущен Chrome) видит не тот каталог → лечится указанием правильного пути/cwd |
| ProcMon: `config.ini` → `NAME NOT FOUND` по «не тому» пути | подтверждён баг пути к конфигу; правильный путь виден в трейсе |
| ProcMon: `config.ini` → `SUCCESS`, но провайдер всё равно не грузится | конфиг читается — причина глубже (само-проверка/инициализация); зафиксировать и остановиться |
| ProcMon: `config.ini` не трогается вовсе, есть `AppPath` `NAME NOT FOUND` | продукт ждёт путь из реестра; тогда вопрос — официальный способ его задать (см. док) |
| Копия `config.ini` в каком-то каталоге «оживила» провайдер | найден ожидаемый путь — записать точный каталог |

## 4. Коммит и отчёт

```powershell
git add docs/minicsp-snapshots/windows-config-trace tools/windows/reverse-enum-test.ps1
git commit -m "docs: windows config.ini load trace for Mini CSP activation"
git push origin HEAD:main
```
В отчёте владельцу: какой именно путь к `config.ini` ожидает `nmcades.exe`/`capi20`
и подтвердилось ли, что «нашёл конфиг → провайдер ожил». Этого достаточно, чтобы
решить, как корректно разложить Mini CSP в нашем launcher (правильный каталог /
рабочая директория), без изменений в самих компонентах CryptoPro.

## 5. Если что-то идёт не так

- Любой шаг, ощущающийся как обход защиты/анализ уязвимостей, — **стоп, спросить
  владельца**. По задумке здесь только наблюдение и штатная настройка.
- Все временные копии файлов — **вернуть как было** после проверки.
- Не коммитить бинарные артефакты CryptoPro и приватные ключи/сертификаты.
