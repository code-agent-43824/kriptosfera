Криптосфера

    
    Примерное техническое задание на продукт и подробное техническое задание на MVP

    
    Статус документа

    Документ фиксирует продуктовую и техническую постановку проекта «Криптосфера» на текущем этапе. Это рабочее техническое задание для проектирования, разработки MVP и дальнейшего развития продукта.

    Документ состоит из двух частей:

    Примерное техническое задание на продукт целиком.

    Подробное техническое задание на MVP.

    

    
    
    
    
    Часть 1. Примерное техническое задание на продукт целиком

    
    1. Назначение продукта

    «Криптосфера» — это технология и услуга по созданию брендированных десктопных бизнес-приложений для российских компаний, чьи клиенты работают с электронной подписью, прежде всего с УКЭП.

    Приложение представляет собой специализированную браузерную оболочку, подготовленную для работы с веб-сервисом заказчика и клиентскими криптографическими компонентами. Пользователь скачивает один файл, запускает его, вставляет USB-токен и начинает работу с сайтом заказчика без ручной настройки обычного браузера, браузерных расширений, нативных плагинов и системного криптопровайдера.

    Основная цель продукта — снизить сложность входа пользователя в рабочий сценарий с электронной подписью и дать заказчику более контролируемую клиентскую среду.

    
    
    2. Проблема, которую решает продукт

    Сейчас работа с УКЭП в веб-сервисах обычно происходит через обычный пользовательский браузер. Такой подход создает несколько проблем.

    Пользователь смешивает рабочее и личное пространство: в одном браузере могут находиться личные аккаунты, сторонние расширения, сохраненные пароли, личная почта, мессенджеры и рабочий сервис, где подписываются юридически значимые документы.

    Рабочее место требует предварительной настройки: установки криптопровайдера, браузерного расширения, нативного плагина, драйверов и дополнительных компонентов. Для обычного пользователя это сложный процесс, а для заказчика — источник обращений в поддержку.

    Веб-сервис заказчика зависит от состояния пользовательского окружения: версии браузера, установленных расширений, состояния CryptoPro CSP, прав пользователя в операционной системе, доступности токена и сертификатов.

    Обычные браузеры не предназначены специально для российских криптографических сценариев. В перспективе это может быть важно для ГОСТ TLS и других задач, хотя ГОСТ TLS не входит в MVP.

    
    
    3. Целевая модель продукта

    Продукт состоит из технологической платформы и процесса сборки брендированных приложений под конкретных заказчиков.

    Для каждого заказчика может выпускаться отдельная сборка приложения. Такая сборка содержит название, иконку, стартовый URL, набор разрешенных доменов, настройки окна, диагностический режим и встроенные компоненты, необходимые для работы с электронной подписью.

    С точки зрения пользователя приложение должно выглядеть как самостоятельное рабочее приложение конкретной компании: банка, оператора ЭДО, удостоверяющего центра, оператора отчетности или иного бизнес-сервиса.

    С технической точки зрения приложение является автономным запускаемым файлом, который при первом запуске разворачивает внутреннюю среду в пользовательский каталог и запускает встроенный браузерный runtime с заданной конфигурацией.

    
    
    4. Целевые заказчики

    Потенциальные заказчики продукта:

    Операторы электронного документооборота.

    Удостоверяющие центры.

    Банки и интернет-банки для бизнеса.

    Операторы отчетности.

    B2B-платформы, где документы или операции подписываются УКЭП.

    Государственные и окологосударственные сервисы.

    Крупные корпоративные заказчики, которым требуется стандартизированная клиентская среда для работы с КЭП.

    Продукт не предполагается позиционировать как массовый браузер общего назначения. Основная модель — разработка, адаптация и сопровождение брендированного приложения под конкретного заказчика.

    
    
    5. Бизнес-модель

    Базовая монетизация:

    Разработка брендированного приложения под заказчика.

    Выпуск новых сборок приложения.

    Техническая поддержка.

    Адаптация под изменения сайта заказчика.

    Сопровождение при изменениях операционных систем, браузерных runtime и криптографических компонентов.

    Развитие дополнительной функциональности.

    Потенциальные дополнительные функции:

    Отслеживание срока действия сертификатов.

    Предупреждение пользователя о скором истечении сертификата.

    Полуавтоматический перевыпуск сертификатов.

    Расширенная диагностика рабочего места.

    Формирование диагностического отчета для поддержки.

    Поддержка нескольких сервисов одного заказчика.

    Централизованная настройка политик для корпоративных клиентов.

    Более глубокая изоляция рабочего пространства от обычного браузера пользователя.

    
    
    6. Основные пользовательские сценарии продукта

    
    6.1. Первый запуск приложения

    Пользователь получает ссылку на приложение от заказчика.

    Пользователь скачивает один исполняемый файл.

    Пользователь запускает файл двойным кликом.

    Приложение тихо разворачивает внутренние компоненты в пользовательский каталог.

    Приложение запускает встроенный браузерный runtime.

    Открывается сайт заказчика или стартовый экран приложения.

    Пользователь вставляет USB-токен.

    Веб-сервис внутри приложения получает доступ к компонентам электронной подписи.

    Пользователь выполняет рабочий сценарий.

    
    
    6.2. Повторный запуск

    Пользователь запускает тот же файл двойным кликом.

    Приложение проверяет, что нужная версия внутренних компонентов уже развернута.

    Приложение не выполняет повторную распаковку без необходимости.

    Приложение запускает сайт заказчика в ранее подготовленном профиле.

    Пользователь продолжает работу.

    
    
    6.3. Диагностика

    Пользователь или специалист поддержки открывает диагностический экран.

    Приложение показывает состояние ключевых компонентов.

    При ошибке приложение показывает, какой компонент недоступен.

    В расширенной версии приложение может формировать диагностический отчет.

    
    
    6.4. Выпуск новой версии

    Разработчик готовит новую сборку приложения.

    В сборку входят обновленные компоненты, конфигурация и версия приложения.

    Заказчик распространяет новую сборку своим каналом.

    Пользователь скачивает и запускает новую версию приложения.

    Приложение разворачивает новую версию внутренней среды рядом с предыдущей или заменяет ее согласно политике сборки.

    
    
    
    7. Целевые платформы

    Первая приоритетная платформа — Windows.

    macOS должна развиваться параллельно как proof of concept кроссплатформенности. Цель macOS-направления — убедиться, что архитектура не становится заведомо Windows-only.

    Linux рассматривается как перспективное направление, но не входит в первый этап.

    
    
    8. Общая техническая архитектура продукта

    Базовая архитектура продукта:

    Single-file launcher / bootstrapper  - встроенный сжатый payload  - тихое разворачивание payload в пользовательский каталог  - проверка версии и целостности развернутых компонентов  - подготовка пользовательских настроек для native messaging  - запуск встроенного Chromium runtime  - передача параметров запуска  - ведение диагностического лога  - обработка ошибок запуска

    Развернутая внутренняя среда:

    Kriptosfera data directory  apps/    customer/      version/        chromium/        extensions/        native-host/        cryptopro/        diagnostics/        config/  profiles/    customer/  logs/

    Основной runtime приложения — готовый бинарный Chromium runtime, поставляемый внутри payload. Сборка Chromium из исходников не является частью процесса разработки MVP и базового продукта.

    
    
    9. Базовый технологический стек

    Рекомендуемый стек для продукта:

    Launcher / bootstrapper: Go.

    Браузерный runtime: готовая бинарная сборка Chromium или совместимого Chromium-based runtime.

    Конфигурация заказчика: JSON или YAML.

    Диагностический интерфейс: статический HTML/JavaScript, позднее возможно расширение до React-интерфейса.

    Сборка и упаковка: GitHub Actions, PowerShell, Go build tooling, собственный механизм встраивания payload.

    E2E-проверки разработки: Playwright / Node.js.

    Хранение профиля: отдельный user-data-dir для каждой сборки/заказчика.

    Electron, CEF, Qt WebEngine, Flutter и Tauri могут рассматриваться как альтернативы или направления для отдельных экспериментов, но не являются основным рекомендуемым стеком для MVP.

    
    
    10. Требования к поставке

    Приложение должно поставляться пользователю как один запускаемый файл.

    Пользовательский опыт должен быть следующим:

    Пользователь скачивает один файл.

    Пользователь запускает один файл двойным кликом.

    Пользователь не видит установщик.

    Пользователь не выбирает каталог установки.

    Пользователь не проходит wizard с шагами «Далее», «Установить», «Готово».

    Пользователь не устанавливает зависимости вручную.

    После запуска открывается рабочее окно приложения или стартовый экран подготовки рабочей среды.

    Внутренне приложение может распаковывать компоненты в пользовательский каталог: AppData на Windows, Application Support на macOS или аналогичный пользовательский каталог на Linux в перспективе.

    Допустим branded splash screen при первом запуске, если подготовка среды занимает заметное время. Splash screen должен восприниматься как часть запуска приложения, а не как установщик.

    
    
    11. Требования к браузерной оболочке

    Браузерная оболочка должна:

    Открывать заданный URL заказчика.

    Использовать отдельный профиль, не связанный с обычным браузером пользователя.

    Загружать только заранее разрешенные расширения.

    Поддерживать запуск в режиме отдельного приложения или ограниченного браузерного окна.

    Не предлагать пользователю сделать себя браузером по умолчанию.

    Не показывать first-run экраны обычного браузера.

    Не использовать личный профиль пользователя из обычного браузера.

    Позволять ограничивать навигацию заданным списком доменов.

    Позволять открывать диагностическую страницу.

    Писать диагностические логи запуска.

    
    
    12. Требования к работе с электронной подписью

    Приложение должно обеспечивать работу веб-страницы с клиентскими компонентами электронной подписи.

    В целевой модели приложение должно содержать:

    Расширение CryptoPro CAdES Browser Plug-in.

    Нативный компонент CryptoPro Browser Plugin.

    CryptoPro CSP Lite или необходимые библиотеки криптопровайдера.

    Конфигурацию native messaging.

    Стабильную привязку расширения к native host.

    Возможность работы с сертификатом на USB-токене.

    На первом этапе проверяется работа без установленного в системе CryptoPro CSP.

    Использование системного CryptoPro CSP как fallback не входит в текущую проверку.

    
    
    13. Требования к брендированию

    Для каждой сборки приложения должны задаваться:

    Название приложения.

    Иконка приложения.

    Стартовый URL.

    Список разрешенных доменов.

    Название заказчика.

    Версия сборки.

    Идентификатор приложения.

    Параметры диагностического режима.

    Параметры окна.

    Желательно, чтобы эти параметры задавались декларативно через конфигурационный файл сборки.

    Пример конфигурации:

    {  "appId": "ru.kriptosfera.customer.demo",  "productName": "Криптосфера Demo",  "customerName": "Demo Customer",  "version": "0.1.0",  "startUrl": "https://www.cryptopro.ru/sites/default/files/products/cades/demopage/cades_bes_sample.html",  "allowedOrigins": ["https://www.cryptopro.ru"],  "profileName": "demo",  "windowMode": "app",  "diagnosticsEnabled": true}

    
    
    14. Требования к диагностике

    Приложение должно иметь диагностический режим.

    Минимальная диагностика:

    Версия приложения.

    Версия payload.

    Путь к развернутым компонентам.

    Путь к профилю браузера.

    Запущенный URL.

    Наличие браузерного runtime.

    Наличие расширения.

    Наличие native host.

    Наличие manifest native messaging.

    Наличие криптографических компонентов.

    Результат запуска браузерного runtime.

    Ошибки запуска.

    Ссылка на тестовую страницу или встроенный тест.

    В будущих версиях диагностика может быть расширена проверкой видимости токена, сертификатов, срока действия сертификатов и формированием отчета для поддержки.

    
    
    15. Требования к безопасности и изоляции

    Приложение должно создавать отдельную рабочую среду, не смешанную с личным браузером пользователя.

    Базовые требования:

    Отдельный user-data-dir.

    Отдельное хранилище cookies и local storage.

    Запрет загрузки неразрешенных расширений.

    Запуск только с заданной конфигурацией.

    Возможность ограничить навигацию списком разрешенных доменов.

    Логи не должны содержать закрытые ключи, PIN-коды и иные секреты.

    Приложение не должно запрашивать у пользователя PIN-код самостоятельно, если это делает штатный криптографический компонент.

    Приложение не должно пытаться извлекать закрытые ключи с токена.

    Канал распространения готового приложения обеспечивает заказчик.

    
    
    16. Требования к обновлению

    На первом этапе встроенный механизм автообновления не требуется.

    Новые версии компонентов поставляются в новой сборке приложения.

    Приложение должно поддерживать версионирование внутреннего payload, чтобы новая версия могла корректно развернуться в пользовательский каталог и не конфликтовать с предыдущей.

    
    
    17. Требования к сборке

    Сборочный процесс должен:

    Собирать launcher.

    Подготавливать payload.

    Включать в payload браузерный runtime.

    Включать в payload расширение, native host, криптографические компоненты и конфигурацию.

    Сжимать payload.

    Встраивать payload в единый запускаемый файл.

    Формировать итоговый артефакт сборки.

    Выкладывать артефакт как файл для скачивания.

    Сборка не должна требовать компиляции Chromium из исходников.

    
    
    18. Требования к тестированию продукта

    Должны быть предусмотрены следующие уровни проверки:

    Проверка запуска launcher.

    Проверка распаковки payload.

    Проверка повторного запуска без повторной распаковки.

    Проверка запуска Chromium runtime.

    Проверка открытия стартового URL.

    Проверка отдельного профиля браузера.

    Проверка загрузки расширения.

    Проверка native messaging manifest.

    Проверка диагностической страницы.

    Ручная проверка работы с токеном.

    Ручная проверка тестовой страницы CryptoPro.

    Автоматические тесты должны покрывать все, что не требует физического USB-токена.

    

    
    
    
    Часть 2. Подробное техническое задание на MVP

    
    1. Цель MVP

    Цель MVP — доказать, что можно создать автономное Windows-приложение, поставляемое пользователю как один запускаемый файл, которое открывает тестовую страницу CryptoPro CAdES Browser Plug-in и обеспечивает работу сценария электронной подписи с сертификатом на Рутокене без установленного в системе CryptoPro CSP.

    Параллельно должен быть выполнен macOS proof of concept, подтверждающий, что выбранная архитектура не является заведомо Windows-only.

    
    
    2. Основной критерий успеха MVP

    MVP считается успешным, если пользователь на чистой или специально подготовленной тестовой Windows-среде без установленного системного CryptoPro CSP может:

    Скачать один .exe файл.

    Запустить его двойным кликом.

    Увидеть открывшееся приложение.

    Открыть внутри приложения тестовую страницу CryptoPro.

    Подключить Рутокен с сертификатом.

    Добиться успешной проверки компонентов на странице CryptoPro.

    Выполнить тестовую операцию подписи.

    Референсная страница:

    https://www.cryptopro.ru/sites/default/files/products/cades/demopage/cades_bes_sample.html

    
    
    3. Границы MVP

    
    3.1. Входит в MVP

    Windows-приложение в виде одного запускаемого файла.

    Собственный launcher / bootstrapper на Go.

    Встроенный сжатый payload.

    Тихая распаковка payload в пользовательский каталог.

    Запуск готового Chromium runtime из payload.

    Открытие тестовой страницы CryptoPro.

    Использование отдельного профиля Chromium.

    Загрузка CryptoPro browser extension.

    Настройка native messaging для связи расширения с нативным компонентом.

    Подключение нативного компонента CryptoPro Browser Plugin.

    Подключение CryptoPro CSP Lite или необходимых библиотек.

    Проверка работы с Рутокеном.

    Минимальный диагностический экран или диагностический режим.

    Логирование запуска.

    Повторный запуск без повторной ручной настройки.

    macOS proof of concept на той же общей архитектурной идее.

    
    
    3.2. Не входит в MVP

    ГОСТ TLS.

    Linux-версия.

    Полноценный брендированный интерфейс.

    Автообновление.

    Централизованная система управления политиками.

    Удаленный сбор диагностических отчетов.

    Полуавтоматический перевыпуск сертификатов.

    Мониторинг срока действия сертификатов.

    Поддержка всех возможных токенов.

    Собственная сборка Chromium из исходников.

    Полноценный установщик.

    Выбор каталога установки пользователем.

    Использование системного CryptoPro CSP как fallback.

    
    
    
    4. Основной технический стек MVP

    
    4.1. Windows MVP

    Язык launcher: Go.

    Браузерный runtime: готовая бинарная сборка Chromium или совместимого Chromium-based runtime.

    Конфигурация: JSON.

    Диагностика: статический HTML/JavaScript или текстовый диагностический экран.

    Упаковка: собственный single-file bootstrapper с встроенным payload.

    Автоматизация сборки: GitHub Actions.

    Скрипты сборки: PowerShell.

    Автоматические проверки: Go tests, PowerShell smoke scripts, Playwright для проверок браузерного поведения.

    
    
    4.2. macOS PoC

    Проверяется применимость общей схемы: launcher, payload, Chromium runtime, отдельный профиль, стартовый URL.

    Полная функциональность электронной подписи на macOS не является обязательным критерием первого Windows MVP, но должна быть исследована.

    Цель macOS PoC — выявить архитектурные blockers как можно раньше.

    
    
    
    5. Архитектура Windows MVP

    Итоговый файл:

    KriptosferaDemo.exe

    При первом запуске файл должен:

    Определить путь пользовательского каталога.

    Определить версию встроенного payload.

    Проверить, развернута ли такая версия payload.

    Если версия не развернута — тихо распаковать payload.

    Создать структуру каталогов.

    Подготовить профиль Chromium.

    Подготовить конфигурацию native messaging.

    Запустить Chromium runtime с нужными параметрами.

    Передать стартовый URL.

    Записать лог запуска.

    При повторном запуске файл должен:

    Проверить наличие уже развернутого payload.

    Проверить версию payload.

    Не распаковывать payload повторно без необходимости.

    Обновить конфигурацию native messaging, если требуется.

    Запустить Chromium runtime.

    
    
    6. Предлагаемая структура каталогов MVP

    Для Windows:

    LOCALAPPDATA/Kriptosfera/  apps/    demo/      0.1.0/        chromium/          chrome.exe or chromium.exe        extensions/          cryptopro-cades/        native-host/          cryptopro-native-host.exe          manifest.json        cryptopro/          plugin/          csp-lite/        diagnostics/          diagnostics.html        config/          app-config.json  profiles/    demo/  logs/    launcher.log

    Конкретные имена каталогов могут быть изменены в процессе разработки, но должны оставаться предсказуемыми и диагностируемыми.

    
    
    7. Конфигурация MVP

    MVP должен использовать конфигурационный файл сборки.

    Минимальная конфигурация:

    {  "appId": "ru.kriptosfera.demo",  "productName": "Криптосфера Demo",  "version": "0.1.0",  "startUrl": "https://www.cryptopro.ru/sites/default/files/products/cades/demopage/cades_bes_sample.html",  "allowedOrigins": ["https://www.cryptopro.ru"],  "profileName": "demo",  "diagnosticsEnabled": true,  "chromiumArgs": [    "--no-first-run",    "--no-default-browser-check"  ]}

    Конфигурация должна использоваться launcher при подготовке окружения и запуске Chromium.

    
    
    8. Требования к launcher

    Launcher должен быть отдельным Go-приложением.

    Функции launcher:

    Определить текущую ОС.

    Проверить совместимость ОС с текущей сборкой.

    Определить пользовательский каталог приложения.

    Создать корневой каталог Kriptosfera.

    Проверить состояние развернутого payload.

    Распаковать payload при первом запуске или смене версии.

    Проверить наличие исполняемого файла Chromium.

    Подготовить профиль Chromium.

    Подготовить native messaging manifest.

    При необходимости записать настройки native messaging в HKCU.

    Сформировать список аргументов запуска Chromium.

    Запустить Chromium.

    Записать лог запуска.

    Обработать ошибки и показать понятное сообщение.

    Launcher не должен:

    Показывать установщик.

    Запрашивать каталог установки.

    Требовать прав администратора.

    Изменять системный браузер пользователя.

    Использовать профиль обычного браузера пользователя.

    Устанавливать расширения в обычный браузер пользователя.

    
    
    9. Требования к встроенному payload

    Payload должен содержать все компоненты, необходимые для первого запуска приложения.

    Минимальный состав payload:

    Chromium runtime.

    CryptoPro CAdES Browser Plug-in extension.

    Нативный компонент CryptoPro Browser Plugin.

    CryptoPro CSP Lite или необходимые библиотеки.

    Конфигурация native messaging.

    Диагностическая страница.

    Конфигурация приложения.

    Payload должен иметь версию и контрольную сумму.

    Launcher должен уметь определить, совпадает ли развернутый payload с встроенным.

    
    
    10. Требования к запуску Chromium

    Chromium должен запускаться:

    Из каталога payload.

    С отдельным user-data-dir.

    С отключением first-run экранов.

    С отключением предложения стать браузером по умолчанию.

    С загрузкой нужного расширения.

    С ограничением набора расширений.

    Со стартовым URL из конфигурации.

    Примерный набор аргументов:

    --user-data-dir=<profile-dir>--load-extension=<cryptopro-extension-dir>--disable-extensions-except=<cryptopro-extension-dir>--app=<start-url>--no-first-run--no-default-browser-check--enable-logging

    Итоговый набор аргументов уточняется в ходе технической проверки.

    
    
    11. Требования к расширению CryptoPro

    В MVP необходимо проверить:

    Может ли расширение CryptoPro CAdES Browser Plug-in быть загружено как unpacked extension.

    Сохраняется ли стабильный extension ID.

    Можно ли зафиксировать extension ID через manifest key или иной допустимый механизм.

    Видит ли тестовая страница CryptoPro загруженное расширение.

    Может ли расширение обращаться к native host.

    Не блокирует ли Chromium запуск расширения в выбранном режиме.

    Если расширение невозможно корректно загрузить как unpacked extension, должен быть зафиксирован blocker и предложен альтернативный путь.

    
    
    12. Требования к native messaging

    MVP должен проверить и реализовать пользовательскую регистрацию native messaging host без прав администратора.

    На Windows требуется:

    Подготовить manifest native messaging host.

    Указать в manifest путь к native host из каталога payload.

    Указать allowed origins для extension ID.

    Зарегистрировать manifest в HKCU.

    Проверить, что Chromium видит native host.

    Проверить обмен сообщениями между расширением и native host.

    Регистрация в HKCU должна выполняться launcher автоматически при запуске.

    
    
    13. Требования к CryptoPro CSP Lite / криптографическим компонентам

    В MVP необходимо проверить:

    Можно ли использовать поставляемые компоненты без системной установки CryptoPro CSP.

    Видит ли нативный компонент нужные криптографические библиотеки из payload.

    Видит ли связка сертификат на Рутокене.

    Может ли тестовая страница выполнить операцию подписи.

    Если для работы потребуется дополнительная настройка переменных окружения или путей поиска библиотек, она должна выполняться launcher автоматически и только в контексте запуска приложения.

    
    
    14. Требования к работе с Рутокеном

    Первичный тестовый токен — Рутокен с сертификатом, пригодным для проверки на тестовой странице CryptoPro.

    MVP должен подтвердить:

    Токен виден криптографическому стеку.

    Сертификат доступен для выбора.

    Операция подписи инициируется со страницы.

    PIN-код, если требуется, запрашивается штатным компонентом.

    Подпись успешно создается в тестовом сценарии.

    Приложение не должно самостоятельно извлекать ключи, PIN-коды или секретные данные.

    
    
    15. Диагностика MVP

    В MVP должна быть реализована минимальная диагностика.

    Формат диагностики может быть простым. Допустимы:

    HTML-страница diagnostics.html.

    Текстовый лог launcher.log.

    Отдельный режим запуска с параметром –diagnostics.

    Минимальные диагностические данные:

    Версия launcher.

    Версия payload.

    Путь к пользовательскому каталогу приложения.

    Путь к Chromium runtime.

    Путь к профилю.

    Путь к extension.

    Путь к native host manifest.

    Состояние записи HKCU для native messaging.

    Стартовый URL.

    Фактические аргументы запуска Chromium.

    Ошибки распаковки.

    Ошибки запуска Chromium.

    Желательно добавить кнопку или инструкцию «Открыть тестовую страницу CryptoPro», если диагностика реализована как HTML-страница.

    
    
    16. Требования к логированию MVP

    Launcher должен писать лог в пользовательский каталог.

    Лог должен содержать:

    Дату и время запуска.

    Версию приложения.

    Версию payload.

    Проверку наличия развернутого payload.

    Результат распаковки.

    Результат подготовки native messaging.

    Команду запуска Chromium без секретных данных.

    Код ошибки, если запуск не удался.

    Лог не должен содержать PIN-коды, закрытые ключи, секретные значения и персональные данные сверх технически необходимых путей и параметров.

    
    
    17. Требования к ошибкам и сообщениям пользователю

    Если запуск невозможен, приложение должно показать короткое понятное сообщение.

    Примеры ошибок:

    Не удалось развернуть внутренние компоненты.

    Не найден браузерный runtime.

    Не удалось подготовить профиль приложения.

    Не удалось запустить рабочее окно.

    Не удалось подготовить компонент связи с расширением.

    Для MVP достаточно простого системного message box или минимального окна ошибки.

    
    
    18. Требования к сборке MVP

    Сборка должна выполняться в GitHub Actions.

    Pipeline MVP:

    Checkout репозитория.

    Установка Go.

    Сборка launcher.

    Подготовка payload directory.

    Загрузка или подключение заранее выбранного Chromium runtime.

    Копирование extension, native host, криптографических компонентов и config.

    Сжатие payload.

    Встраивание payload в launcher.

    Формирование итогового файла KriptosferaDemo.exe.

    Публикация файла как build artifact.

    Сборка Chromium из исходников не выполняется.

    
    
    19. Требования к репозиторию MVP

    Рекомендуемая структура репозитория:

    kriptosfera/  cmd/    kriptosfera-launcher/      main.go  internal/    bootstrap/    payload/    registry/    chromium/    config/    logging/  payload-template/    extensions/    native-host/    cryptopro/    diagnostics/    config/  build/    build-windows.ps1    prepare-payload.ps1    embed-payload.ps1  tests/    smoke/    playwright/  .github/    workflows/      build-windows.yml  README.md

    
    
    20. Этапы разработки MVP

    
    Этап 1. Single-file bootstrapper без Chromium

    Цель: доказать модель одного файла.

    Результат:

    Один .exe.

    Внутри встроен тестовый payload.

    При запуске payload тихо распаковывается в пользовательский каталог.

    Повторный запуск не распаковывает payload заново.

    Пишется лог.

    Запускается тестовый внутренний executable или открывается тестовый файл.

    
    
    Этап 2. Запуск Chromium runtime

    Цель: доказать запуск встроенного браузерного runtime.

    Результат:

    Payload содержит Chromium runtime.

    Launcher запускает Chromium из пользовательского каталога.

    Chromium открывает тестовую страницу.

    Используется отдельный профиль.

    Не показываются first-run экраны.

    
    
    Этап 3. Загрузка extension

    Цель: проверить загрузку CryptoPro extension.

    Результат:

    Extension входит в payload.

    Chromium запускается с этим extension.

    Тестовая страница видит extension или фиксируется конкретная причина, почему не видит.

    Проверяется стабильность extension ID.

    
    
    Этап 4. Native messaging

    Цель: проверить связь extension с native host.

    Результат:

    Launcher создает manifest.

    Launcher регистрирует manifest в HKCU.

    Extension может обратиться к native host.

    В логах видно успешное подключение или конкретная ошибка.

    
    
    Этап 5. CryptoPro components и Рутокен

    Цель: проверить полный сценарий подписи.

    Результат:

    Нативный компонент видит криптографические библиотеки из payload.

    Тестовая страница видит компоненты.

    Рутокен виден.

    Сертификат доступен.

    Тестовая подпись создается.

    
    
    Этап 6. Минимальная диагностика

    Цель: сделать MVP проверяемым и удобным для отладки.

    Результат:

    Есть диагностическая HTML-страница или режим.

    Есть лог запуска.

    Ошибки можно локализовать без отладчика.

    
    
    Этап 7. macOS PoC

    Цель: проверить переносимость архитектурной идеи.

    Результат:

    Есть минимальный launcher для macOS.

    Payload разворачивается в пользовательский каталог.

    Запускается Chromium runtime.

    Открывается тестовая страница.

    Фиксируются отличия и ограничения macOS.

    
    
    
    21. Автоматические проверки MVP

    Автоматически проверяется:

    Сборка launcher.

    Наличие payload.

    Корректность config.

    Распаковка payload.

    Повторный запуск.

    Наличие Chromium runtime.

    Запуск Chromium.

    Открытие стартового URL.

    Наличие логов.

    Наличие диагностической страницы.

    Ручная проверка требуется для:

    Подключения Рутокена.

    Проверки видимости сертификата.

    Ввода PIN-кода.

    Создания тестовой подписи.

    Проверки поведения CryptoPro Browser Plugin.

    
    
    22. Критерии приемки MVP

    MVP принимается, если выполнены следующие критерии:

    Итоговый артефакт Windows-сборки представляет собой один .exe файл.

    Файл запускается двойным кликом без установщика.

    Приложение не требует прав администратора для базового запуска.

    Приложение не требует ручной установки системного CryptoPro CSP.

    Приложение не требует ручной установки браузерного расширения.

    Приложение не требует ручной установки нативного плагина пользователем.

    Payload разворачивается автоматически в пользовательский каталог.

    Chromium runtime запускается из развернутого payload.

    Открывается тестовая страница CryptoPro.

    Используется отдельный профиль браузера.

    Extension загружается внутри встроенного Chromium.

    Native messaging настроен автоматически.

    Тестовая страница видит необходимые компоненты.

    Рутокен с сертификатом доступен для тестового сценария.

    Тестовая операция подписи выполняется успешно.

    Повторный запуск работает без повторной ручной настройки.

    Есть лог запуска.

    Есть минимальная диагностика.

    Есть macOS PoC, подтверждающий принципиальную переносимость подхода или фиксирующий конкретные ограничения.

    
    
    23. Основные технические риски MVP

    Открытые технические риски:

    Возможность загрузки CryptoPro extension в выбранном Chromium runtime.

    Стабильность extension ID.

    Корректная работа native messaging с расширением из embedded/развернутого окружения.

    Возможность запуска native host из пользовательского каталога.

    Возможность использования CryptoPro CSP Lite без системной установки CSP.

    Видимость токена и сертификата из автономной поставки.

    Особенности macOS при доступе к токенам и запуске внешних компонентов.

    Размер итогового .exe и время первого запуска.

    Поведение антивирусов и систем защиты при запуске самораспаковывающегося приложения.

    Корректное удаление или сосуществование нескольких версий payload.

    
    
    24. Необходимые результаты MVP

    По итогам MVP должны быть получены:

    Репозиторий с исходным кодом launcher.

    Сборочный pipeline.

    Один Windows .exe артефакт.

    Инструкция для разработчика по сборке.

    Инструкция по ручному тесту с Рутокеном.

    Логика развертывания payload.

    Конфигурационная модель приложения.

    Диагностический режим.

    Протокол проверки на тестовой странице CryptoPro.

    Список выявленных ограничений.

    macOS proof of concept.

    
    
    25. Краткая итоговая формулировка MVP

    MVP «Криптосферы» — это один запускаемый Windows-файл, который без установщика и без прав администратора разворачивает в пользовательский каталог автономную рабочую среду, запускает встроенный Chromium runtime, открывает тестовую страницу CryptoPro CAdES Browser Plug-in и обеспечивает тестовое создание электронной подписи с сертификатом на Рутокене без установленного в системе CryptoPro CSP.
