Криптосфера

    
    Описание идеи и первичная концепция продукта

    
    1. Краткое описание

    «Криптосфера» — это технология и услуга по созданию брендированных десктопных бизнес-приложений для российских компаний, чьи клиенты работают с электронной подписью, прежде всего с УКЭП.

    Приложение представляет собой специализированную браузерную оболочку, заранее подготовленную для работы с электронной подписью. Пользователь скачивает один файл, запускает его, вставляет USB-токен и сразу начинает работать с веб-сервисом заказчика без ручной настройки браузера, расширений, плагинов и системного криптопровайдера.

    Целевой результат — снизить сложность входа пользователя в рабочий сценарий с КЭП и дать заказчику более контролируемую клиентскую среду.

    
    
    2. Проблема

    Сейчас работа с УКЭП в веб-сервисах обычно происходит через обычный пользовательский браузер. Это неудобно и для пользователя, и для компании, предоставляющей сервис.

    Основные проблемы текущего подхода:

    Смешение рабочего и личного пространства.Пользователь работает с ЭДО, банком, УЦ или другим бизнес-сервисом в том же браузере, где открыта личная почта, мессенджеры, личные аккаунты, сторонние расширения и сохраненные пароли.

    Сложная настройка рабочего места.Для работы с КЭП пользователю обычно нужно установить криптопровайдер, браузерное расширение, нативный плагин, драйверы или компоненты для работы с токеном. Для неподготовленного пользователя это сложный и плохо предсказуемый процесс.

    Зависимость от состояния пользовательского окружения.Работа сервиса зависит от версии браузера, установленных расширений, состояния CryptoPro CSP, наличия плагина, прав пользователя в операционной системе и других внешних факторов.

    Высокая нагрузка на поддержку.Значительная часть обращений может быть связана не с самим веб-сервисом, а с настройкой клиентского рабочего места.

    Ограниченная поддержка российских криптографических сценариев в обычных браузерах.Большинство привычных браузеров не рассчитаны на полноценную работу с российской криптографией, включая сценарии с ГОСТ TLS. При этом ГОСТ TLS не входит в первый MVP «Криптосферы» и рассматривается как возможное направление развития.

    
    
    3. Идея решения

    Идея заключается в создании технологии, позволяющей быстро собирать отдельные десктопные приложения под конкретные российские веб-сервисы, использующие электронную подпись.

    Такое приложение должно выглядеть для пользователя как самостоятельное приложение конкретной компании: банка, оператора ЭДО, удостоверяющего центра, оператора отчетности или другой организации.

    Внутри приложение является тонким специализированным браузером, который:

    открывает заранее заданный URL;

    содержит необходимые компоненты для работы с КЭП;

    не требует ручной установки расширений и плагинов;

    не требует системной установки CryptoPro CSP на первом этапе проверки;

    работает в изолированном рабочем окружении;

    может быть брендирован под конкретного заказчика.

    Пользовательский сценарий должен быть максимально простым: скачать файл, запустить приложение, вставить USB-токен, начать работу.

    
    
    4. Целевая аудитория заказчиков

    Потенциальные заказчики «Криптосферы»:

    операторы электронного документооборота;

    удостоверяющие центры;

    банки и интернет-банки для бизнеса;

    операторы отчетности;

    B2B-платформы, где документы или операции подписываются УКЭП;

    государственные и окологосударственные сервисы;

    крупные корпоративные заказчики, которым требуется стандартизированная клиентская среда для работы с КЭП.

    Продукт предполагается продавать не как массовый браузер для конечных пользователей, а как услугу по созданию, адаптации и сопровождению брендированного приложения под конкретного заказчика.

    
    
    5. Целевой пользовательский сценарий

    Базовый сценарий выглядит следующим образом:

    Пользователь получает от компании ссылку на приложение.

    Пользователь скачивает один файл.

    Пользователь запускает этот файл двойным кликом.

    Приложение открывает сайт заказчика.

    При необходимости приложение разворачивает внутренние компоненты в пользовательский каталог, например AppData на Windows.

    Пользователь вставляет USB-токен с сертификатом УКЭП.

    Веб-сервис внутри приложения получает доступ к необходимым криптографическим компонентам.

    Пользователь выбирает сертификат и выполняет рабочее действие: подписывает документ, подтверждает операцию, входит в личный кабинет или выполняет иной сценарий с КЭП.

    Для пользователя не должно быть отдельного этапа установки и настройки CryptoPro CSP, браузерного расширения или нативного плагина.

    
    
    6. Техническая гипотеза

    Техническая гипотеза проекта: можно собрать автономное десктопное приложение, внутри которого будут поставляться браузерный движок, компоненты CryptoPro Browser Plugin и библиотеки CryptoPro CSP Lite, достаточные для работы веб-страницы с электронной подписью без системной установки CryptoPro CSP.

    Возможные технологические основы для оболочки:

    Chromium / CEF;

    Electron;

    Qt WebEngine;

    иной подходящий браузерный движок или desktop-фреймворк.

    Внутри приложения предполагается поставлять:

    браузерный движок;

    расширение CryptoPro CAdES Browser Plug-in;

    нативный компонент CryptoPro Browser Plugin;

    CryptoPro CSP Lite или необходимые библиотеки криптопровайдера;

    конфигурационные файлы и манифесты для связи браузерного расширения с нативным компонентом;

    минимальные средства самодиагностики.

    На первом этапе важно доказать именно автономный сценарий: работа без установленного в системе CryptoPro CSP. Возможность использовать системный CryptoPro CSP как fallback может быть рассмотрена позже, но не является целью первичной проверки.

    
    
    7. Платформы

    Первая целевая платформа для MVP — Windows.

    macOS должна проверяться параллельно как proof of concept кроссплатформенности. Цель macOS-направления на этом этапе — убедиться, что выбранный технический подход не создает заведомо Windows-only архитектуру.

    Linux рассматривается как перспективное направление, но не входит в первый этап.

    
    
    8. Требование к поставке приложения

    Требование «один файл» трактуется с точки зрения пользовательского опыта, а не как жесткое ограничение внутренней реализации.

    Пользователь должен скачать и запустить один файл. После запуска приложение может распаковывать необходимые компоненты в пользовательский каталог, например:

    AppData на Windows;

    Application Support или аналогичный каталог на macOS;

    пользовательский каталог приложения на Linux в перспективе.

    Важно, чтобы пользователь не выполнял ручную установку зависимостей и не настраивал компоненты отдельно.

    
    
    9. Границы MVP

    MVP должен доказать возможность автономной работы приложения с УКЭП на Windows.

    В MVP входит:

    Сборка Windows-приложения, запускаемого пользователем как один файл.

    Открытие заданного URL внутри специализированной браузерной оболочки.

    Подключение встроенного расширения CryptoPro CAdES Browser Plug-in.

    Работа нативного компонента CryptoPro Browser Plugin внутри поставки приложения.

    Использование CryptoPro CSP Lite или соответствующих библиотек без системной установки CryptoPro CSP.

    Проверка работы с сертификатом на USB-токене.

    Проверка на Рутокенах как первичных тестовых устройствах.

    Минимальная диагностика состояния компонентов.

    Параллельный macOS proof of concept для проверки переносимости подхода.

    В MVP не входит:

    ГОСТ TLS;

    полноценная поддержка Linux;

    поддержка всех возможных токенов;

    централизованное управление политиками;

    корпоративное массовое развертывание;

    автообновление;

    удаленный сбор диагностических отчетов;

    полуавтоматический перевыпуск сертификатов;

    мониторинг срока действия сертификатов;

    использование системного CryptoPro CSP как fallback.

    
    
    10. Референсный тест

    В качестве первичной референсной страницы используется тестовая страница CryptoPro CAdES Browser Plug-in:

    https://www.cryptopro.ru/sites/default/files/products/cades/demopage/cades_bes_sample.html

    Критерий успеха: страница внутри приложения должна показать успешное состояние всех необходимых компонентов и позволить выполнить тестовый сценарий создания подписи.

    Если этот тест успешно проходит, следующим шагом можно проверять реальные веб-сервисы: Диадок, сайты удостоверяющих центров, банки, операторы ЭДО и другие сервисы, использующие аналогичный клиентский стек.

    
    
    11. Работа с токенами

    Первичная проверка проводится на Рутокенах.

    При этом архитектура не должна искусственно завязываться только на Рутокен. Базовое предположение: если корректно заработает связка браузерной оболочки, CryptoPro Browser Plugin и CryptoPro CSP Lite, то другие совместимые токены также могут заработать при условии поддержки соответствующим криптографическим стеком.

    Проверка других токенов может быть вынесена в последующий этап compatibility testing.

    
    
    12. Диагностика

    Даже в MVP желательно предусмотреть минимальную диагностическую информацию для пользователя или поддержки.

    Приложение должно уметь показывать состояние ключевых компонентов:

    загружено ли браузерное расширение;

    доступен ли нативный компонент CryptoPro Browser Plugin;

    доступен ли криптопровайдер из поставки приложения;

    виден ли подключенный USB-токен;

    видны ли сертификаты;

    может ли быть выполнена тестовая операция подписи.

    На первом этапе это может быть простая диагностическая панель или отдельный экран без сложной системы логирования.

    
    
    13. Критерии приемки MVP

    MVP можно считать успешным, если выполняются следующие условия:

    Пользователь скачивает один файл приложения для Windows.

    Пользователь запускает приложение без установки системного CryptoPro CSP.

    Пользователь не устанавливает отдельно браузерное расширение и нативный плагин.

    Приложение самостоятельно разворачивает необходимые компоненты в пользовательский каталог, если это требуется.

    Приложение открывает тестовую страницу CryptoPro CAdES Browser Plug-in.

    Тестовая страница видит необходимые компоненты.

    Приложение видит сертификат на Рутокене.

    Пользователь может выполнить тестовую операцию подписи.

    После закрытия и повторного запуска приложение продолжает работать без повторной ручной настройки.

    Есть минимальная диагностика состояния компонентов.

    Есть отдельный macOS proof of concept, подтверждающий принципиальную переносимость подхода.

    
    
    14. Бизнес-модель

    «Криптосфера» рассматривается как собственный продукт, pet project и потенциальный стартап.

    Основная монетизация:

    разработка брендированного приложения под конкретного заказчика;

    сопровождение приложения;

    выпуск обновлений;

    адаптация под изменения сайта заказчика;

    поддержка новых версий операционных систем, браузерных движков и криптографических компонентов;

    расширение функциональности.

    Возможные будущие функции:

    отслеживание срока действия сертификатов;

    предупреждение пользователя о скором истечении сертификата;

    полуавтоматический перевыпуск сертификатов;

    расширенная диагностика рабочего места;

    формирование диагностического отчета для поддержки;

    автообновление приложения;

    поддержка нескольких сервисов одного заказчика;

    централизованная настройка политик для корпоративных клиентов;

    более глубокая изоляция рабочего пространства от обычного браузера пользователя.

    
    
    15. Основные риски, ограничения и принятые решения

    На текущем этапе остаются вопросы, которые требуют отдельной технической проверки:

    Можно ли корректно встроить CryptoPro Browser Plugin и его нативный компонент в выбранную браузерную оболочку.

    Можно ли обеспечить работу CryptoPro CSP Lite без системной установки и регистрации CSP в Windows.

    Как именно браузерное расширение будет связываться с нативным компонентом в изолированной поставке.

    Какие ограничения накладывают Windows и macOS на запуск, подпись приложения, notarization, sandboxing и доступ к USB-токенам.

    Можно ли добиться одинаковой архитектурной модели на Windows и macOS.

    Отдельно зафиксированы принятые решения, которые не считаются открытыми вопросами для MVP:

    Реальные сайты ЭДО, банков, УЦ и других сервисов не рассматриваются как жестко завязанные на один конкретный браузер или одну ОС. Базовое предположение: если они работают в разных браузерах и на разных ОС, то основной риск находится не в самих сайтах, а в корректности встроенного клиентского криптографического окружения.

    Использование системного CryptoPro CSP как fallback не входит в текущую проверку.

    Обновление встроенных компонентов не требует отдельного механизма внутри MVP. Новые версии компонентов поставляются вместе с новой сборкой приложения.

    Распространение готового приложения обеспечивает сам заказчик. Вопросы канала распространения, доверенной доставки и защиты от подмены на стороне доставки не входят в первый MVP.

    
    
    16. Краткая формулировка

    «Криптосфера» — это технология сборки автономных десктопных приложений для работы с российскими веб-сервисами, использующими УКЭП. Приложение представляет собой специализированный браузер с заранее встроенными компонентами для работы с электронной подписью. Пользователь скачивает один файл, запускает его, вставляет USB-токен и работает с сайтом заказчика без ручной настройки браузера, плагинов и системного криптопровайдера. Для заказчиков это услуга по созданию и сопровождению брендированного приложения, снижающего сложность подключения пользователей к сервисам с КЭП.
